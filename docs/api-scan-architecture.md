# api-scan: Описание решения, рамки MVP и архитектура

> Версия: 1.1  
> Статус: MVP (с Billing Service)  
> Дата: 2026-03-17

---

## 1. Цели продукта

**Миссия:** исключить ручной ввод паспортных данных в B2B-секторе за счёт облачного OCR-распознавания и бесшовной интеграции с 1С.

**Ключевые бизнес-цели:**

- Сократить время заведения карточки клиента/сотрудника с 3–5 минут до ~5 секунд.
- Снизить количество ошибок в юридически значимых документах.
- Предоставить функционал уровня Enterprise малому и среднему бизнесу по модели оплаты по факту использования или предоплаты.

**Целевая аудитория:**

- Абонентские отделы (провайдеры связи, ЖКХ)
- Финансовый сектор (МФО, ломбарды, кредитные кооперативы)
- HoReCa и услуги (отели, турфирмы, аренда авто/оборудования)
- HR-департаменты с массовым наймом
- Аутсорсинговые бухгалтерские и юридические компании

---

## 2. Рамки MVP

### Входит в MVP

- Облачный API-сервис распознавания паспортов РФ (OCR + нормализация данных)

- Биллинг: транзакционный учёт в рублях, резервирование и списание средств
- Идемпотентность повторных запросов (24 часа)
- Версионированный REST API (`/v1/`)
- Личный кабинет: регистрация, управление API-ключами, баланс, история событий аккаунта, покупка пакетов через ЮКассу
- OCR: Yandex Vision (primary) + VK Vision (fallback при недоступности)

### Не входит в MVP (оставлено на следующие итерации)

- Проверка паспорта на действительность по реестру МВД (публичный реестр закрыт МВД в 2023 году; единственный актуальный канал — СМЭВ, подключение к которому требует отдельного проекта)
- Собственная on-premise OCR-модель
- Enterprise-тарифы с SLA-гарантиями и приоритетной очередью
- Расширение для 1С (разрабатывается отдельным треком)
- Асинхронная модель распознавания (`GET /v2/recognize/{request_id}`) и вебхуки по завершении обработки — запланированы в v2
- Распознавание страницы с пропиской (требует отдельного изображения, работа со штампами существенно сложнее печатного текста; адрес передаётся в Dadata для стандартизации и сопоставления с ФИАС)
- Распознавание документов, кроме паспорта РФ

### Критерии успеха MVP

1. Успешное распознавание паспорта РФ (печатные данные) с точностью ≥ 95% на качественных фото.
2. Проведение первой коммерческой транзакции через систему биллинга.
3. Время ответа API (end-to-end, включая OCR) ≤ 7 секунд в 95-м перцентиле.

## 3. Нефункциональные требования

| Параметр | Требование |
|---|---|
| Доступность | ≥ 99.5% SLA |
| Latency (P95) | ≤ 7 сек (включая OCR) |
| Пиковая нагрузка | 100 RPS (горизонтальное масштабирование Serverless Container) |
| Хранение изображений | Только в RAM, на диск не пишется |
| Локализация данных | Yandex Cloud, регион РФ |
| Шифрование | HTTPS TLS 1.2+ |
| Соответствие | 152-ФЗ: сервис является обработчиком ПДн, оператором — клиент |

## 4. Архитектура сервиса

### 4.1. Обзор

Сервис полностью размещается в **Yandex Cloud (РФ)**. Клиент (1С-расширение) обращается по HTTPS с API-ключом. Изображение документа обрабатывается исключительно в памяти — на диск не сохраняется.

```
1С Расширение
     │  POST /v1/recognize  (HTTPS + API Key + Idempotency-Key?)
     ▼
API Gateway  ──── 401 Unauthorized (невалидный ключ)
     │
     ▼
Idempotency Cache (Redis)
     │  cache hit → снять резерв, вернуть кэшированный результат
     │  cache miss → продолжить
     ▼
Billing: проверка баланса 
     |     → 402 Payment Required (баланс <= 0 руб.)
     |     → резервирование суммы операции (PENDING)
     │
     ▼
Core Orchestrator (прямой вызов, без очереди)
     ├── OCR: Yandex Vision v2 (primary)
     │         ├── Structured model (passport) → entities
     │         ├── Недостаточно полей       → fallback → generic page model + парсинг
     │         └── 5xx / таймаут            → fallback → VK Vision v1 (generic OCR)
     └── Нормализатор: ФИО, даты, серия/номер, код подразделения
     │
     ▼
Billing: фиксация результата
     ├── confidence ≥ порога у хотя бы одного провайдера → COMMITTED
     ├── оба провайдера вернули 5xx / таймаут            → ROLLBACK + 503
     └── оба провайдера вернули confidence < порога      → ROLLBACK + 422
     │
     ▼
JSON-ответ (с request_id и блоком confidences) → 1С

── Yandex Message Queue используется только в асинхронном v2 ──
```

### 4.2. Компоненты

#### API Gateway
- Yandex Cloud Application Load Balancer + WAF (защита от DDoS и OWASP Top 10)
- Ответственность: TLS-терминация, rate-limiting, маршрутизация по версии (`/v1/`, `/v2/`)
- **Аутентификация по API-ключу:** при первой валидации ключ проверяется по bcrypt-хэшу в БД; результат (`api_key_hash → org_id`) кэшируется в Redis с TTL 5 минут. Последующие запросы проверяются по кэшу — bcrypt не вызывается на каждом запросе. При отзыве ключа запись в Redis инвалидируется немедленно.
- **Rate-limiting:** 10 RPS на API-ключ по умолчанию; лимит настраивается через тариф. При недоступности Redis — возвращаем `503 Service Unavailable` (осознанное решение: fallback в PostgreSQL при 100 RPS создаёт риск каскадного отказа).

#### Billing Service (отдельный микросервис)
**Архитектура:** Отдельный сервис с собственной БД `billing_db`. Полная автономность. Детальное описание в `billing-architecture.md`.

**Event Sourcing:** Источник правды — `billing_events` (immutable лог). `balance_snapshots` — read-only кэш.

**Двойная система:**
- **Prepaid:** Списание с предоплаченного баланса при активной подписке
- **Pay-as-you-go:** Списание с рублёвого баланса при отсутствии подписки

**Двухфазная операция:**
- **Reserve:** Проверка баланса → создание резерва в `reservations` (TTL 5 мин)
- **Commit/Rollback:** Фиксация в `billing_events` или удаление резерва

**Жизненный цикл подписки:** `active` → `grace_period` (7 дней, prepaid работает) → `expired`

**Взаимодействие:** Cabinet (управление), Orchestrator (Reserve/Commit), ЮКасса (платежи)

#### Idempotency Cache
- Хранилище: Yandex Memory Store (Redis), TTL 24 часа
- Ключ идемпотентности: опциональный заголовок `Idempotency-Key` (строка 10–64 символа)
- **Логика:** Проверка кэша происходит до создания резерва. При cache hit — сразу возвращается закэшированный ответ без обращения к Billing и OCR.
- При недоступности Redis: пропускаем проверку, обрабатываем как новый запрос (редкий дубль лучше недоступности)

#### Yandex Message Queue
Используется в асинхронной модели v2 (за рамками MVP). В синхронном v1 оркестратор вызывается напрямую без очереди.

В v2 очередь будет выполнять роль буфера между Gateway и Orchestrator: выравнивать пиковую нагрузку, защищать от превышения квот OCR-провайдеров и гарантировать доставку задачи при временной недоступности оркестратора.

Интерфейс очереди (`queue.Publish` / `queue.Consume`) абстрагирован в коде — конкретная реализация (YMQ, Kafka, SQS) задаётся через конфигурацию.

#### Core Orchestrator
- Yandex Serverless Container (stateless Docker-образ, запускается в любом Kubernetes / Serverless)
- В v1 вызывается напрямую из Gateway (без очереди). В v2 будет читать задачи из Message Queue.
- **OCR провайдеры:** Yandex Vision v2 (primary) + VK Vision (fallback). Yandex Vision v2 поддерживает модели распознавания (`passport`, `driver-license-front` и др.) и имеет внутренний fallback: если structured model не вернула достаточно полей — автоматически пробуется generic `page` модель с текстовым парсингом. При `5xx`, таймауте или ошибке API — внешний fallback на VK Vision. Провайдеры используются поочерёдно, не параллельно.
- **Порог confidence:** настраивается через конфигурацию (стартовое значение — 0.80). Поля с confidence ниже порога возвращаются как `null`, confidence по каждому полю включается в ответ.
- **Circuit Breaker:** при N последовательных ошибках от провайдера за M секунд — провайдер временно исключается из ротации на период T, запросы идут напрямую к fallback.
- **PDF:** конвертируется в изображение в изолированном процессе (sandbox) перед передачей в OCR. Plain text и метаданные PDF не используются.
- Изображение хранится в RAM в течение одного запроса, после — уничтожается.

**Абстракции в коде** — все внешние зависимости скрыты за интерфейсами:

```
interfaces/
  ocr_provider   — YandexVisionV2 | YandexVision (legacy) | VKVision | Mock
  queue          — YMQ | Kafka | SQS (для v2)
  cache          — Redis | NoopCache (fallback)
  storage        — PostgreSQL interface
```

Конкретные реализации задаются через dependency injection и переменные окружения. Смена провайдера — изменение конфига, не кода.

#### Нормализатор
- Разбивка строки ФИО на Фамилию / Имя / Отчество
- Форматирование дат к стандарту `ДД.ММ.ГГГГ`
- Валидация серии (4 цифры), номера (6 цифр), кода подразделения (формат `XXX-XXX`)
- Поле `registration_address` возвращается как `null` — зарезервировано для распознавания прописки (за рамками MVP)
- Формирует блок `confidences` с уровнем уверенности по каждому полю (значения 0.0–1.0 из ответа OCR-провайдера)

#### Базы данных
- **`api_scan_main`** (PostgreSQL): организации, пользователи, API-ключи, история событий аккаунта. Используется Gateway и Cabinet.
- **`billing_db`** (PostgreSQL): счета (`accounts`), подписки, биллинг-события, балансы. Изолированная БД Billing Service. Детали в `billing-architecture.md`.

### 4.3. Версионирование API и модели взаимодействия

- Все эндпоинты версионируются с первого релиза: `/v1/recognize`
- При введении breaking change выпускается `/v2/` при живом `/v1/`
- Период поддержки устаревшей версии — не менее 6 месяцев (расширения обновляются медленно)

#### Модель взаимодействия v1 — синхронная

Клиент отправляет запрос и ждёт ответа в рамках одного HTTP-соединения. Таймаут на стороне клиента — не менее 15 секунд.

**Запрос:**
```http
POST /v1/recognize
Content-Type: multipart/form-data
X-Api-Key: <api_key>

file: <image_file>
document_type: passport_rf  # опционально, default: passport_rf
```

Поддерживаемые типы документов (MVP — только `passport_rf`):
- `passport_rf` — Паспорт РФ (первоначально)
- В будущем: `snils`, `inn`, `drivers_license`, `passport_foreign` и др.

**Ответ:**
```json
{
  "request_id": "req_01J...",
  "document_type": "passport_rf",
  "last_name": "Иванов",
  "first_name": "Иван",
  "middle_name": "Иванович",
  "birth_date": "01.01.1990",
  "series": "4510",
  "number": "123456",
  "issue_date": "01.01.2010",
  "issued_by": "ГУ МВД РОССИИ ПО Г. МОСКВЕ",
  "division_code": "770-001",
  "registration_address": null,
  "confidences": {
    "last_name": 0.999,
    "first_name": 0.998,
    "middle_name": 0.997,
    "birth_date": 0.996,
    "series": 0.967,
    "number": 0.967,
    "issue_date": 0.999,
    "issued_by": 0.999,
    "division_code": 0.991
  }
}
```

Каждый успешный ответ содержит `request_id` — уникальный идентификатор запроса. Поле `document_type` в ответе повторяет переданный (или дефолтный) тип документа.

#### Модель взаимодействия v2 — асинхронная (будущее)

Планируется в следующей итерации после MVP. Клиент отправляет документ и немедленно получает `request_id` со статусом `pending`. Результат получает одним из трёх способов:

- **Поллинг:** `GET /v2/recognize/{request_id}` — периодически запрашивает статус (`pending` → `processing` → `done` / `error`).
- **Вебхук:** клиент заранее регистрирует `callback_url` через `POST /v2/webhooks`; сервис делает POST на этот URL по завершении обработки каждой задачи.
- **Синхрон:** `POST /v2/recognize/sync` — для обратной совместимости с клиентами, предпочитающими синхронную модель.

Все три механизма используют единый `request_id`. `request_id` из v1 и v2 находятся в едином пространстве имён — результат операции из v1 доступен через v2-эндпоинты.

#### Расширение: поддержка разных типов документов (будущее)

В MVP поддерживается только `passport_rf`. В следующих итерациях планируется поддержка других документов (СНИЛС, ИНН, водительское удостоверение и др.).

**Вариант A: Явное указание типа (текущий)**
Клиент передаёт `document_type` в запросе. Просто, но требует знания типа документа заранее.

**Вариант B: Автоматическая классификация (будущее)**
Предварительный шаг — лёгкая модель классификации (например, на базе EfficientNet или аналогичной) определяет тип документа перед OCR:
- Стоимость классификации — минимальная (дешёвле OCR в 10-100 раз)
- Точность классификации > 99% для стандартных документов
- При неуверенности — запрос к клиенту или fallback к "паспорт по умолчанию"

**Совместимость:** API v1 уже готов к расширению — поле `document_type` присутствует в запросе и ответе. Добавление новых типов документов — это новые значения `document_type` и новые нормализаторы, не breaking change.

### 4.4. Биллинг: тарифная модель (Billing Service)

**Важно:** Детальное описание Billing Service в `billing-architecture.md`. Здесь — краткий обзор для контекста.

Поддерживаются два типа списания:
- **`pay_as_you_go`** — оплата по факту с рублёвого баланса
- **`prepaid`** — подписка с предоплаченным пакетом операций

#### Структура тарифов

**`tariffs`** — справочник (immutable):
| Поле | Тип | Описание |
|---|---|---|
| `id` | `smallserial PK` | числовой идентификатор |
| `code` | `varchar` | читаемый код (`free`, `basic`, `pro`) |
| `name` | `varchar` | отображаемое название |

**`tariff_versions`** — версии с ценами и сроками:
| Поле | Тип | Описание |
|---|---|---|
| `id` | `serial PK` | |
| `tariff_id` | `smallint FK → tariffs.id` | |
| `valid_from/until` | `timestamptz` | период действия версии |
| `duration_days` | `int` | длительность подписки |
| `base_price_rub` | `numeric` | стоимость подписки |
| `prepaid_amount_rub` | `numeric` | сумма предоплаченных кредитов |

**`tariff_service_prices`** — цены операций:
| Поле | Тип | Описание |
|---|---|---|
| `tariff_version_id` | `int FK` | |
| `service_id` | `varchar` | тип операции (`passport_rf`) |
| `included_price_rub` | `numeric` | цена в предоплаченном пакете |
| `overage_price_rub` | `numeric` | цена сверх пакета |

#### Баланс организации

Хранится в `balance_snapshots` (кэш) и пересчитывается из `billing_events`:
- `real_balance_rub` — рублёвый баланс для pay-as-you-go
- `prepaid_balance_rub` — остаток предоплаченных кредитов

**Приоритет списания:** При активной подписке — сначала `prepaid_balance_rub`, затем (если закончился) `real_balance_rub` по цене `overage_price_rub`.

### 4.5. История событий аккаунта

Таблица `account_events` — append-only лог всех значимых действий по организации:

| Поле | Тип | Описание |
|---|---|---|
| `id` | `bigserial PK` | |
| `org_id` | `int FK` | организация |
| `event_type` | `varchar` | тип события (см. ниже) |
| `payload` | `jsonb` | детали события |
| `created_at` | `timestamptz` | |
| `actor_id` | `int nullable FK` | пользователь ЛК, если применимо |

Типы событий (`event_type`):

- `tariff_changed` — смена тарифа: `{ from_tariff_id, to_tariff_id }`
- `balance_topped_up` — пополнение рублёвого баланса: `{ amount_rub, order_id }`
- `api_key_created` — выпуск нового ключа: `{ key_id, name }`
- `api_key_revoked` — отзыв ключа: `{ key_id }`
- `user_invited` — добавление пользователя в ЛК
- `user_removed` — удаление пользователя

Запись в таблицу производится внутри той же транзакции, что и основное изменение — чтобы событие не потерялось при откате.

### 4.6. Личный кабинет

Веб-интерфейс для управления аккаунтом. Функциональность:

- **Регистрация и аутентификация** организации (email + пароль, с подтверждением email)
- **API-ключи:** генерация именованных ключей, просмотр даты создания и последнего использования, отзыв с немедленной инвалидацией в API Gateway
- **Баланс и тариф:** текущий рублёвый баланс, остаток предоплаченных операций (если тариф `prepaid`), текущий тариф, кнопка смены тарифа
- **Покупка пакетов:** выбор пакета → оплата через ЮКассу
- **История событий:** лог из `account_events` с фильтрацией по типу и дате

### 4.7. Интеграция с ЮКассой

Пополнение баланса реализовано в **Billing Service**. Кабинет инициирует платёж через Billing API, Billing взаимодействует с ЮКассой напрямую.

Детали флоу, вебхуков и безопасности — в `billing-architecture.md` (раздел «Интеграция с Cabinet Service» и «Пополнение через ЮКассу»).

## 5. Организация кодовой базы

Проект использует монорепозиторий `api-scan`. Все сервисы, общие пакеты и инфраструктура находятся в одном репозитории — это упрощает рефакторинг интерфейсов и синхронизацию изменений между сервисами.

```
api-scan/
├── services/
│   ├── orchestrator/     # Core Orchestrator
│   ├── billing/          # Billing Service
│   └── cabinet/          # Личный кабинет (backend + frontend)
├── pkg/
│   ├── ocr/              # интерфейс OcrProvider + адаптеры YandexVision, VKVision, Mock
│   ├── queue/            # интерфейс Queue + адаптеры YMQ, Mock
│   ├── cache/            # интерфейс Cache + адаптеры Redis, NoopCache
│   └── storage/          # общие типы и интерфейсы БД
├── infra/
│   └── terraform/        # вся инфраструктура (staging + production)
├── migrations/
│   ├── main/             # миграции api_scan_main (Gateway, Cabinet)
│   └── billing/          # миграции billing_db (Billing Service)
└── docs/                 # архитектурные документы
```

Каждый сервис в `services/` собирается в независимый Docker-образ. Общие интерфейсы из `pkg/` импортируются сервисами — конкретные реализации (OCR-провайдер, очередь, кэш) задаются через dependency injection и переменные окружения.

Расширение для 1С разрабатывается в отдельном репозитории `1c-scan` — другой стек, другой цикл релизов, другая команда.

## 6. Безопасность и соответствие 152-ФЗ

- **Stateless processing:** изображения не пишутся на диск, не передаются в хранилища. Обработка только в RAM в рамках одного запроса.
- **Роли субъектов ПДн:** клиент — оператор персональных данных; api-scan — обработчик. Это закрепляется в публичной оферте и DPA (Data Processing Agreement).
- **Yandex Vision:** изображения передаются в API Яндекса. Необходимо подписать DPA с Яндексом и прописать это в условиях для клиентов.
- **Шифрование:** TLS 1.2+ для всех соединений. Данные в БД шифруются средствами Yandex Managed PostgreSQL.
- **API-ключи:** хранятся в виде bcrypt-хэша. На горячем пути (каждый API-запрос) проверяются через Redis-кэш — bcrypt вызывается только при первой валидации нового ключа. При отзыве — кэш инвалидируется немедленно.
