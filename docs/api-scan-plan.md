# api-scan: План реализации MVP

> Версия: 1.1  
> Статус: актуальный  
> Дата: 2026-03-17

Задачи сгруппированы по этапам. Каждый этап — завершённый рабочий срез, который можно тестировать независимо. Задачи внутри этапа выполняются последовательно, если не указано иное.

Организационные и инфраструктурные задачи, которые нужно закрыть до начала разработки.

### 0-5. Организовать монорепозиторий и настроить CI/CD

**Что сделать:**
Создать Git-репозиторий `api-scan` с монорепо-структурой. Организовать каталоги сервисов, общих пакетов и инфраструктуры. Настроить базовый CI: линтер, тесты, сборка Docker-образов. Настроить CD: автодеплой в `staging` при мерже в `main`.

Структура репозитория:

```
api-scan/
├── services/
│   ├── orchestrator/     # Core Orchestrator
│   ├── billing/          # Billing Service
│   └── cabinet/          # Личный кабинет (backend + frontend)
├── pkg/
│   ├── ocr/              # интерфейс OcrProvider + адаптеры YandexVision, VKVision, Mock
│   ├── queue/            # интерфейс Queue + адаптеры YMQ, Mock
│   ├── cache/            # интерфейс Cache + адаптеры Redis, NoopCache
│   └── storage/          # общие типы и интерфейсы БД
├── infra/
│   └── terraform/        # вся инфраструктура (staging + production)
├── migrations/
│   ├── main/             # миграции api_scan_main (Gateway, Cabinet)
│   └── billing/          # миграции billing_db (Billing Service)
└── docs/                 # архитектурные документы
```

**Критерии выполнения:**
- Репозиторий создан, структура каталогов соответствует схеме выше и согласована командой.
- Каждый сервис в `services/` собирается в независимый Docker-образ командой из корня репозитория.
- Пакеты из `pkg/` импортируются сервисами без дублирования кода.
- Pull request с любым изменением запускает линтер и тесты затронутых сервисов — красный PR нельзя смержить.
- Мерж в `main` автоматически деплоит изменённые сервисы в `staging` в течение 10 минут.
- Деплой в `production` требует ручного подтверждения.
- Terraform в `infra/terraform/` применяется из CI — прямые изменения через UI Yandex Cloud запрещены.
- Terraform state хранится в Yandex Object Storage, не в репозитории.

## Этап 2. Core Orchestrator и OCR

---

### 2-1. Реализовать интеграцию с Yandex Vision

**Что сделать:**
Определить интерфейс `OcrProvider` и написать адаптер Yandex Vision: принимает байты изображения, отправляет запрос в Vision, возвращает структурированный результат с полями и значениями confidence. Изображение нигде не сохраняется — только в памяти в рамках вызова.

**Критерии выполнения:**
- Адаптер успешно распознаёт тестовый набор из 20 сканов паспортов РФ (чёткие фото).
- Изображение не записывается на диск — проверяется через `lsof` или аналог в тестовом окружении.
- При `5xx`-ответе от Yandex Vision адаптер выбрасывает типизированное исключение `OcrProviderError`.
- Адаптер покрыт unit-тестами с моками HTTP-клиента.

---

### 2-2. Реализовать интеграцию с VK Vision (fallback)

**Что сделать:**
Написать адаптер для VK Vision API по аналогии с Yandex Vision. Оба адаптера реализуют единый интерфейс `OcrProvider`.

**Критерии выполнения:**
- VK Vision адаптер реализует тот же интерфейс `OcrProvider`, что и Yandex Vision.
- При подмене провайдера в конфиге (флаг или env-переменная) сервис использует VK Vision без изменения кода.
- Адаптер покрыт unit-тестами с моками.

---

### 2-3. Реализовать нормализатор паспортных данных

**Что сделать:**
Написать модуль, который принимает сырой текст от OCR и возвращает структурированный объект: фамилия, имя, отчество, дата рождения, серия и номер паспорта, дата выдачи, код подразделения, кем выдан.

**Критерии выполнения:**
- На тестовом наборе из 50 паспортов нового образца (с 1997 года) при качественном фото точность заполнения каждого поля ≥ 95%. Для старых образцов и плохого качества — отдельный замер без гарантий в MVP.
- Даты возвращаются в формате `ДД.ММ.ГГГГ`.
- Серия паспорта возвращается в формате `XXXX` (4 цифры), номер — `XXXXXX` (6 цифр).
- Код подразделения возвращается в формате `XXX-XXX`; значения, не соответствующие формату, возвращаются как `null`.
- Если поле не удалось распознать — оно присутствует в ответе со значением `null`, а не отсутствует.
- Поле `registration_address` всегда присутствует в ответе со значением `null`.
- Ответ содержит блок `confidences` с числовыми значениями 0.0–1.0 для каждого поля.
- При неоднозначности (confidence < порога у отдельных полей, но не у всех) — возвращается частичный результат: уверенные поля заполнены, неуверенные — `null`. Сервис не отказывает целиком из-за одного нераспознанного поля.
- Модуль покрыт unit-тестами с фикстурами реальных OCR-ответов.

---

### 2-4. Реализовать Core Orchestrator

**Зависимость:** Требует задачи **3-2** (Reserve/Commit/Rollback в Billing Service). Можно разрабатывать параллельно, используя мок Billing для задач 2-1..2-3.

**Что сделать:**
Написать основной сервис, который объединяет OCR и нормализатор в единый флоу. Интеграция с Billing Service: Reserve → OCR → Commit/Rollback.

**Флоу:**
1. `POST /accounts/{id}/reserve` — блокировка средств
2. OCR (Yandex Vision → fallback VK Vision)
3. `POST /transactions/{id}/commit` (успех) или `/rollback` (ошибка)

**Критерии выполнения:**
- Сервис поднимается как Yandex Serverless Container (stateless Docker-образ) с параметрами: RAM 2 GB, CPU 2 vCPU, timeout 30 секунд.
- При `5xx`, таймауте или confidence < порога от Yandex Vision — fallback VK Vision.
- Circuit Breaker: 5 ошибок за 60 сек → исключение провайдера на 30 сек.
- PDF конвертируется через `pdftoppm` в sandbox.
- **Секреты:** Токены Yandex/VK Vision хранятся в Yandex Lockbox, монтируются через env. Не в коде, не в Docker-образе.
- **Graceful Shutdown:** При SIGTERM — завершить текущий OCR-запрос, закрыть соединения, только затем выйти.
- Интеграция с Billing Service через HTTP (service token):
  - Reserve перед OCR
  - Commit при успехе
  - Rollback при ошибке OCR или низком confidence
- При недоступности Billing Service — возвращает `503`.
- Интеграционный тест: конец-в-конец с моком OCR и Billing.

## Этап 3. Billing Service (отдельный микросервис)

**Архитектура:** Event Sourcing, отдельная БД `billing_db`. Детали в `billing-architecture.md`.

---

### 3-1. Реализовать базовую структуру сервиса

**Что сделать:**
Создать `services/billing/` с HTTP-сервером, подключением к `billing_db`, миграциями goose.

**Критерии выполнения:**
- Сервис запускается, подключается к `billing_db`.
- Миграции применяются автоматически при старте.
- Health-check эндпоинт `/health` возвращает 200.

---

### 3-2. Реализовать Reserve / Commit / Rollback

**Что сделать:**
Реализовать двухфазную модель:
- `POST /accounts/{id}/reserve` — проверка баланса, создание резерва в `reservations`
- `POST /transactions/{id}/commit` — фиксация: запись в `billing_events`, удаление резерва
- `POST /transactions/{id}/rollback` — откат: удаление резерва

**Логика списания:**
- При активной подписке: `charge_type = 'prepaid'` (списание с `prepaid_balance_rub`)
- При отсутствии подписки или в grace period: `charge_type = 'pay_as_you_go'` (списание с `real_balance_rub`)

**Критерии выполнения:**
- Reserve атомарен: `SELECT FOR UPDATE` + проверка баланса (snapshot + events_since_snapshot - reservations).
- При недостаточном балансе — `402 Payment Required`.
- Commit создаёт запись в `billing_events`, удаляет резерв.
- Rollback просто удаляет резерв.
- Cron-job очищает истёкшие резервы (TTL 5 минут).
- Покрыто unit- и интеграционными тестами (гонки, граничные случаи).

---

### 3-3. Реализовать подписки и тарифы

**Что сделать:**
Реализовать управление подписками:
- `POST /accounts/{id}/subscriptions` — покупка подписки
- `GET /accounts/{id}/subscriptions` — список подписок
- Upgrade/downgrade логика с проратацией

**Критерии выполнения:**
- Покупка списывает стоимость с `real_balance_rub`.
- При upgrade — проратированный бонус на `prepaid_balance_rub`.
- При downgrade — `next_tariff_version_id` сохраняется, применяется при renewal.
- Grace period (7 дней) работает: prepaid списывается, pay-as-you-go требует баланса.

---

### 3-4. Реализовать интеграцию с ЮКассой

**Что сделать:**
Реализовать флоу пополнения баланса:
- `POST /accounts/{id}/topups` — создание платежа, возврат URL ЮКассы
- `POST /webhooks/yookassa` — обработка вебхуков

**Критерии выполнения:**
- При успешном платеже: запись в `billing_events` (type='balance_topup'), обновление `balance_snapshots`.
- Вебхук идемпотентен по `payment_id` (проверка перед обработкой).
- Верификация подписи вебхука обязательна.
- Тестовый режим ЮКассы работает корректно.

---

### 3-5. Реализовать cron jobs

**Что сделать:**
Реализовать фоновые задачи:
- **Recalc balances** (каждые 5 мин): `UPDATE balance_snapshots` на основе `billing_events`
- **Expire subscriptions** (каждые 10 мин): `active` → `grace_period` → `expired`
- **Process downgrades** (каждые 10 мин): применение `pending_downgrade`
- **Cleanup reservations** (каждые 5 мин): удаление истёкших резервов

**Критерии выполнения:**
- Все задачи выполняются без ошибок.
- **Advisory lock:** Перед запуском каждого job — `SELECT pg_try_advisory_lock('job_name')`. Если lock не получен — job пропускается (другой инстанс уже выполняет).
- При пересчёте балансов используется `pg_advisory_xact_lock` — lock автоматически снимается при COMMIT/ROLLBACK.
- Подписки корректно переходят по статусам, остатки prepaid сгорают при expired.

## Этап 4. API Gateway и интеграция с Billing

---

### 4-1. Настроить API Gateway с аутентификацией

**Что сделать:**
Настроить Yandex Application Load Balancer + WAF. Реализовать аутентификацию по заголовку `X-Api-Key`: при первой валидации ключ проверяется bcrypt-хэшем в `api_scan_main`, результат кэшируется в Redis (TTL 5 мин).

**Критерии выполнения:**
- Запрос без заголовка `X-Api-Key` возвращает `401 Unauthorized`.
- Запрос с невалидным ключом возвращает `401 Unauthorized`.
- Запрос с валидным ключом: ключ проверяется по хэшу в БД, результат `api_key_hash → (org_id, billing_account_id)` кэшируется в Redis (TTL 5 мин).
- Rate-limiting: более 30 запросов в секунду возвращают `429`.
- При недоступности Redis — `503 Service Unavailable`.

---

### 4-2. Реализовать эндпоинт `POST /v1/recognize`

**Что сделать:**
Реализовать основной публичный эндпоинт: принимает изображение (JPEG, PNG, PDF) и опциональный параметр `document_type`, запускает полный синхронный флоу обработки, возвращает структурированный JSON.

**Параметры запроса:**
- `file` (обязательный): изображение или PDF
- `document_type` (опциональный): тип документа, default: `passport_rf`. В MVP поддерживается только `passport_rf`.

**Критерии выполнения:**
- Успешный ответ содержит `request_id`, `document_type`, все поля паспорта, блок `confidences` и `registration_address: null`.
- `request_id` уникален для каждого запроса и присутствует в ответе всегда — в том числе при ошибке.
- При неподдерживаемом `document_type` (отличном от `passport_rf` в MVP) возвращает `400 Bad Request`.
- При confidence обоих провайдеров ниже порога — возвращает `422 Unprocessable` с описанием причины; баланс не списывается.
- При технической ошибке обоих провайдеров — возвращает `503`; баланс не списывается.
- Неподдерживаемый формат файла возвращает `415 Unsupported Media Type`.
- Файл размером более 10 МБ возвращает `413 Request Entity Too Large`.
- PDF конвертируется в изображение в изолированном процессе — исходный PDF не передаётся в OCR напрямую.
- Документация API содержит явное указание рекомендуемого таймаута на стороне клиента — не менее 15 секунд.
- Время ответа (end-to-end) ≤ 7 секунд на P95 — проверяется нагрузочным тестом на 20 параллельных запросах с моком OCR.
- Ответ при любой ошибке содержит читаемое сообщение и код ошибки, не стек-трейс.

## Этап 5. Личный кабинет (Cabinet Service)

---

### 5-0. Интеграция с Billing Service

**Что сделать:**
Cabinet Service взаимодействует с Billing Service через HTTP API (service token):
- При регистрации организации — `POST /accounts` в Billing, сохранение `billing_account_id`
- Покупка подписки — `POST /accounts/{id}/subscriptions` в Billing
- Пополнение баланса — `POST /accounts/{id}/topups` в Billing
- Получение баланса — `GET /accounts/{id}/balance` из Billing

**Критерии выполнения:**

### 6-0. Настроить мониторинг и алерты

**Что сделать:**
Настроить Yandex Monitoring для всех сервисов. Определить ключевые метрики и пороги алертов. Настроить канал уведомлений команды.

**Критерии выполнения:**
- В Yandex Monitoring собираются следующие метрики по каждому сервису: latency P50/P95/P99, error rate (4xx, 5xx), RPS, использование CPU и RAM.
- Дополнительные метрики для Core Orchestrator: distribution confidence по полям (гистограмма), частота срабатывания Circuit Breaker, частота fallback на VK Vision.
- Дополнительные метрики для Billing: количество PENDING-транзакций старше 1 минуты, количество ROLLBACK за период.
- Алерты настроены на: error rate > 1% за 5 минут → P1; P95 latency > 7 сек за 5 минут → P2; PENDING-транзакций > 10 → P2; недоступность Redis > 30 сек → P1.
- Канал алертов — Telegram (команда) или Slack; выбор фиксируется командой до начала этапа.
