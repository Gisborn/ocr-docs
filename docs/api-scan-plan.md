# api-scan: План реализации MVP

> Версия: 1.0  
> Статус: черновик  
> Дата: 2025

Задачи сгруппированы по этапам. Каждый этап — завершённый рабочий срез, который можно тестировать независимо. Задачи внутри этапа выполняются последовательно, если не указано иное.

- Terraform в `infra/terraform/` применяется из CI — прямые изменения через UI Yandex Cloud запрещены.
- Terraform state хранится в Yandex Object Storage, не в репозитории.

## Этап 1. База данных и схема

Все сервисы используют одну Managed PostgreSQL. Схема — основа для всей последующей разработки.

---

### 1-1. Развернуть Managed PostgreSQL

**Что сделать:**
Создать инстанс Yandex Managed PostgreSQL в каталогах `staging` и `production`. Настроить резервное копирование, мониторинг, сетевой доступ только из внутренней сети облака.

**Критерии выполнения:**
- PostgreSQL доступен из внутренней сети Yandex Cloud, недоступен из интернета.
- Автоматические бэкапы настроены с retention 7 дней для staging, 30 дней для production.
- Тестовое подключение из staging-окружения успешно.

---

### 1-2. Описать и применить миграции схемы БД

**Что сделать:**
Выбрать и настроить инструмент миграций. Написать начальные миграции для всех таблиц: `organizations`, `users`, `api_keys`, `tariffs`, `tariff_price_plans`, `billing_transactions`, `orders`, `account_events`.

Инструмент: **goose** — поддерживает up/down миграции, advisory locks (защита от одновременного применения несколькими инстансами), интегрируется в Go-код и в CI/CD.

**Критерии выполнения:**
- Все миграции применяются командой `goose up` без ошибок на чистой БД.
- Откат (`goose down`) работает для каждой миграции.
- При одновременном запуске двух инстансов деплоя миграции применяются ровно один раз — advisory lock предотвращает дублирование.
- В `tariffs` присутствует запись с `id=1, code='basic', type='on_demand'`.
- В `tariff_price_plans` присутствует запись для `basic`-тарифа с заполненным `on_demand_price_rub`.
- Внешние ключи и индексы соответствуют схеме из архитектурного документа.
- Миграции применяются автоматически при деплое через CI/CD до запуска новой версии сервиса.

## Этап 2. Core Orchestrator и OCR

---

### 2-1. Реализовать интеграцию с Yandex Vision

**Что сделать:**
Определить интерфейс `OcrProvider` и написать адаптер Yandex Vision: принимает байты изображения, отправляет запрос в Vision, возвращает структурированный результат с полями и значениями confidence. Изображение нигде не сохраняется — только в памяти в рамках вызова.

**Критерии выполнения:**
- Адаптер успешно распознаёт тестовый набор из 20 сканов паспортов РФ (чёткие фото).
- Изображение не записывается на диск — проверяется через `lsof` или аналог в тестовом окружении.
- При `5xx`-ответе от Yandex Vision адаптер выбрасывает типизированное исключение `OcrProviderError`.
- Адаптер покрыт unit-тестами с моками HTTP-клиента.

---

### 2-2. Реализовать интеграцию с VK Vision (fallback)

**Что сделать:**
Написать адаптер для VK Vision API по аналогии с Yandex Vision. Оба адаптера реализуют единый интерфейс `OcrProvider`.

**Критерии выполнения:**
- VK Vision адаптер реализует тот же интерфейс `OcrProvider`, что и Yandex Vision.
- При подмене провайдера в конфиге (флаг или env-переменная) сервис использует VK Vision без изменения кода.
- Адаптер покрыт unit-тестами с моками.

---

### 2-3. Реализовать нормализатор паспортных данных

**Что сделать:**
Написать модуль, который принимает сырой текст от OCR и возвращает структурированный объект: фамилия, имя, отчество, дата рождения, серия и номер паспорта, дата выдачи, код подразделения, кем выдан.

**Критерии выполнения:**
- На тестовом наборе из 50 паспортов нового образца (с 1997 года) при качественном фото точность заполнения каждого поля ≥ 95%. Для старых образцов и плохого качества — отдельный замер без гарантий в MVP.
- Даты возвращаются в формате `ДД.ММ.ГГГГ`.
- Серия паспорта возвращается в формате `XXXX` (4 цифры), номер — `XXXXXX` (6 цифр).
- Код подразделения возвращается в формате `XXX-XXX`; значения, не соответствующие формату, возвращаются как `null`.
- Если поле не удалось распознать — оно присутствует в ответе со значением `null`, а не отсутствует.
- Поле `registration_address` всегда присутствует в ответе со значением `null`.
- Ответ содержит блок `confidences` с числовыми значениями 0.0–1.0 для каждого поля.
- При неоднозначности (confidence < порога у отдельных полей, но не у всех) — возвращается частичный результат: уверенные поля заполнены, неуверенные — `null`. Сервис не отказывает целиком из-за одного нераспознанного поля.
- Модуль покрыт unit-тестами с фикстурами реальных OCR-ответов.

---

### 2-4. Реализовать Core Orchestrator

**Что сделать:**
Написать основной сервис, который объединяет OCR и нормализатор в единый флоу. В v1 вызывается напрямую из Gateway без очереди. Реализовать все интерфейсы (`OcrProvider`, `Cache`) через dependency injection.

**Критерии выполнения:**
- Сервис поднимается как Yandex Serverless Container (stateless Docker-образ) с параметрами: RAM 2 GB, CPU 2 vCPU, timeout 30 секунд.
- При `5xx`, таймауте или confidence < порога от Yandex Vision автоматически выполняется повторный запрос к VK Vision.
- Реализован Circuit Breaker: после 5 последовательных ошибок провайдера за 60 секунд — провайдер исключается из ротации на 30 секунд.
- PDF конвертируется в изображение через `pdftoppm` (poppler-utils) в изолированном процессе с ограниченными правами — ImageMagick не используется. Исходный PDF не передаётся в OCR напрямую.
- Токены Yandex Vision и VK Vision хранятся в Yandex Lockbox и монтируются в контейнер через переменные окружения — не хардкодятся в образе. Ротация — вручную при компрометации.
- Реализован Graceful Shutdown: при получении SIGTERM сервис завершает текущий OCR-запрос, корректно закрывает соединения с БД и Redis, только затем завершает процесс.
- Результат содержит все нормализованные поля, блок `confidences` и `request_id`.
- При технической ошибке (оба провайдера недоступны) — возвращает ошибку с кодом для ROLLBACK биллинга.
- При низком confidence обоих провайдеров — возвращает `422` с кодом для ROLLBACK.
- Интеграционный тест: конец-в-конец запрос с тестовым изображением возвращает корректно заполненный JSON с блоком `confidences`.
- Смена OCR-провайдера через env-переменную не требует изменения кода.

## Этап 3. Billing Service

---

### 3-1. Реализовать проверку баланса и резервирование

**Что сделать:**
Написать внутренний сервис с эндпоинтом `POST /reserve`: принимает `org_id` и `operation_type`, определяет стоимость по прайслисту тарифа, проверяет баланс, создаёт транзакцию со статусом `PENDING`. Реализовать cron-job для cleanup зависших транзакций.

**Критерии выполнения:**
- При достаточном балансе транзакция создаётся со статусом `PENDING`, баланс уменьшается на сумму резерва.
- При нехватке баланса возвращается ошибка `INSUFFICIENT_BALANCE` — транзакция не создаётся.
- Приоритет списания соблюдается: сначала `prepaid_operations_left`, затем `balance_rub`.
- Операция атомарна: одновременные запросы от одной организации не допускают уход баланса в минус (тест с конкурентными запросами).
- Cron-job запускается каждые 5 минут и переводит транзакции в статусе `PENDING` старше 2 минут в `ROLLBACK` — после запуска `PENDING`-транзакций старше 2 минут в БД нет.
- Покрыто unit- и интеграционными тестами, включая граничные случаи (баланс = 0, баланс = стоимость операции).

---

### 3-2. Реализовать фиксацию и откат транзакции

**Что сделать:**
Добавить эндпоинты `POST /commit` и `POST /rollback`: принимают `transaction_id`, переводят транзакцию в статус `COMMITTED` или `ROLLBACK` соответственно.

**Критерии выполнения:**
- `COMMITTED`: транзакция переходит в финальный статус, повторный вызов возвращает ошибку `ALREADY_COMMITTED`.
- `ROLLBACK`: резерв снимается, баланс возвращается к значению до резервирования.
- Нельзя сделать rollback уже закоммиченной транзакции и наоборот.
- Переходы покрыты тестами для всех допустимых и недопустимых комбинаций статусов.

---

### 3-3. Реализовать идемпотентность повторных запросов

**Что сделать:**
Подключить Yandex Memory Store (Redis). Реализовать поддержку заголовка `Idempotency-Key` (строка 10–64 символа): если заголовок передан, проверять наличие ключа в Redis перед резервированием. Реализовать `Cache`-интерфейс с Redis-адаптером и NoopCache-адаптером (fallback при недоступности Redis).

**Критерии выполнения:**
- Повторный запрос с тем же `Idempotency-Key` от той же организации в течение 24 часов возвращает закэшированный результат.
- Баланс при повторном запросе не изменяется — проверяется сравнением `balance_rub` до и после.
- Запрос без заголовка `Idempotency-Key` обрабатывается без проверки идемпотентности.
- При недоступности Redis запрос проходит без ошибки — используется NoopCache (осознанный компромисс).
- TTL ключа в Redis подтверждён командой `TTL <key>` в тестовом окружении.
- При отзыве API-ключа организации — Redis-кэш для него инвалидируется немедленно.

## Этап 4. API Gateway

---

### 4-1. Настроить API Gateway с аутентификацией

**Что сделать:**
Настроить Yandex Application Load Balancer + WAF. Реализовать аутентификацию по заголовку `X-Api-Key`: при первой валидации ключ проверяется bcrypt-хэшем в БД, результат кэшируется в Redis (TTL 5 мин). Запрос обогащается `org_id` и передаётся в Core Orchestrator напрямую.

**Критерии выполнения:**
- Запрос без заголовка `X-Api-Key` возвращает `401 Unauthorized`.
- Запрос с невалидным ключом возвращает `401 Unauthorized`.
- Запрос с валидным ключом передаётся в Core Orchestrator с корректным `org_id`.
- Запрос с недостаточным балансом возвращает `402 Payment Required`.
- Rate-limiting: более 30 запросов в секунду с одного API-ключа возвращают `429 Too Many Requests`.
- При недоступности Redis — сервис возвращает `503 Service Unavailable`. Fallback в PostgreSQL не используется: при 100 RPS переброс нагрузки на БД создаёт риск каскадного отказа. Это осознанное решение — кратковременная недоступность предпочтительнее перегрузки БД.
- После отзыва ключа в ЛК — повторный запрос с этим ключом возвращает `401` без задержки (Redis-кэш инвалидирован).
- WAF настроен: SQL-инъекция в теле запроса блокируется с кодом `403`.
- Все эндпоинты доступны только по HTTPS; HTTP-запросы отклоняются.

---

### 4-2. Реализовать эндпоинт `POST /v1/recognize`

**Что сделать:**
Реализовать основной публичный эндпоинт: принимает изображение (JPEG, PNG, PDF) и опциональный параметр `document_type`, запускает полный синхронный флоу обработки, возвращает структурированный JSON.

**Параметры запроса:**
- `file` (обязательный): изображение или PDF
- `document_type` (опциональный): тип документа, default: `passport_rf`. В MVP поддерживается только `passport_rf`.

**Критерии выполнения:**
- Успешный ответ содержит `request_id`, `document_type`, все поля паспорта, блок `confidences` и `registration_address: null`.
- `request_id` уникален для каждого запроса и присутствует в ответе всегда — в том числе при ошибке.
- При неподдерживаемом `document_type` (отличном от `passport_rf` в MVP) возвращает `400 Bad Request`.
- При confidence обоих провайдеров ниже порога — возвращает `422 Unprocessable` с описанием причины; баланс не списывается.
- При технической ошибке обоих провайдеров — возвращает `503`; баланс не списывается.
- Неподдерживаемый формат файла возвращает `415 Unsupported Media Type`.
- Файл размером более 10 МБ возвращает `413 Request Entity Too Large`.
- PDF конвертируется в изображение в изолированном процессе — исходный PDF не передаётся в OCR напрямую.
- Документация API содержит явное указание рекомендуемого таймаута на стороне клиента — не менее 15 секунд.
- Время ответа (end-to-end) ≤ 7 секунд на P95 — проверяется нагрузочным тестом на 20 параллельных запросах с моком OCR.
- Ответ при любой ошибке содержит читаемое сообщение и код ошибки, не стек-трейс.

## Этап 5. Личный кабинет

---

### 5-1. Реализовать регистрацию и аутентификацию

**Что сделать:**
Реализовать регистрацию организации (email + пароль), подтверждение email, вход и выход. Пароли хранятся в виде bcrypt-хэша.

**Критерии выполнения:**

- Попытка входа с неверным паролем возвращает ошибку без указания, что именно неверно (email или пароль).
- Сессия инвалидируется при выходе — повторное использование токена возвращает `401`.

---

### 5-2. Реализовать управление API-ключами

**Что сделать:**
Реализовать страницу управления ключами: создание именованного ключа, просмотр списка (имя, дата создания, дата последнего использования), отзыв ключа.

**Критерии выполнения:**
- Созданный ключ показывается пользователю один раз полностью; при повторном просмотре — только последние 4 символа.
- Ключ хранится в БД в виде bcrypt-хэша.
- После отзыва ключа запрос к API с этим ключом немедленно возвращает `401` — без задержки на кэш.
- Дата последнего использования обновляется асинхронно: при каждом запросе временная метка записывается в Redis (`key_id → timestamp`), cron-job раз в минуту сбрасывает накопленные значения в PostgreSQL. Точность до минуты для отображения в ЛК достаточна; прямая запись в БД на каждый запрос не используется.
- Организация может иметь не более 10 активных ключей одновременно.

### 6-0. Настроить мониторинг и алерты

**Критерии выполнения:**

- Дополнительные метрики для Core Orchestrator: distribution confidence по полям (гистограмма), частота срабатывания Circuit Breaker, частота fallback на VK Vision.

- Канал алертов — Telegram (команда) или Slack; выбор фиксируется командой до начала этапа.
